# 방법론 평가

검증 방법론에 대한 비판적 검토와 학술 연구 결과를 다룹니다.

---

## Q13: 이 프로젝트의 검증 방법론에 대한 비판과 학술 연구 결과는?

**질문 시각**: 2026-01-05

**답변**:

"vocab_size나 config dimension 비교만으로는 불충분하고, 실제 토큰 중복률이나 weight 값 비교가 더 중요하다"는 의견이 제기되었습니다. 이에 대해 학술 연구를 조사한 결과를 공유합니다.

학술적으로 가장 신뢰할 수 있는 검증 방법은 무엇일까요? arXiv:2502.00706v1에 발표된 "Model Provenance Testing for Large Language Models" 논문에 따르면, 600개 이상의 모델에서 검증된 결과 출력 행동 기반 통계 분석이 가장 신뢰할 수 있는 방법입니다. 동일 프롬프트에 대한 출력 토큰 일치율을 통계 분석하는 black-box output similarity 방법이 90-95% precision과 80-90% recall을 달성했습니다. Config 비교는 불충분한데, fine-tuning은 architecture를 보존하기 때문입니다. Weight 비교는 유효하지만 white-box 접근이 필요합니다.

Config나 dimension 비교에는 한계가 있습니다. config.json 비교의 한계는 fine-tuning이 architecture를 보존하므로 동일한 dimension을 가진 모델이 파생작일 수 있다는 점입니다. vocab_size 비교의 경우, fine-tuning 시 tokenizer를 그대로 사용하므로 vocab_size 일치가 오히려 파생 증거가 될 수 있습니다.

그렇다면 현재 프로젝트 방법론의 유효성은 어떨까요? 현재 프로젝트의 로직은 "vocab_size가 불일치하면 from scratch 지지"입니다. vocab_size가 다르면 tokenizer 재학습이 필요하므로 from scratch의 강력한 증거가 됩니다. 이 로직은 유효합니다. 반면 vocab_size가 같으면 추가 검증이 필요한데, token 중복률과 merge rules를 분석해야 합니다. 따라서 vocab_size 불일치를 기반으로 한 Solar, A.X-K1, VAETKI, K-EXAONE의 판정은 여전히 유효합니다. HyperCLOVAX처럼 vocab_size가 유사한 경우에는 실제 토큰 중복률 분석이 추가로 필요합니다.

Yi-Llama 논란 사례도 참고할 만합니다. 01.AI의 Yi-34B 모델이 Meta Llama에서 파생되었다는 의혹이 있었습니다. Architecture가 Llama와 동일한 구조였고, tensor names도 Llama 형식을 그대로 사용했습니다. 01.AI는 이를 "oversight(실수)"라고 인정했습니다. 하지만 weight 복사 증거는 없었고 독립 학습 주장을 유지했습니다. EleutherAI의 분석에 따르면, Yi는 독립적으로 학습되었으며 Llama architecture 채택은 업계 표준 관행입니다.

방법론 개선 방향을 정리하면, vocab_size 비교에 실제 토큰 중복률 계산을 추가하고, config dimension 비교에 출력 행동 유사도 분석을 추가하면 됩니다. Architecture 비교는 유효한데, architecture가 다르면 weight 재사용이 불가능하기 때문입니다.

비판의 타당성을 평가하면, "config dimension 비교만으로 불충분"하다는 주장은 부분적으로 맞습니다. 같은 architecture에서 fine-tuning이 가능하기 때문입니다. "실제 토큰 중복률이 더 중요"하다는 주장은 맞는데, 특히 vocab_size가 유사할 때 그렇습니다. "weight 비교가 더 중요"하다는 주장은 부분적으로 맞지만, 학술 연구에서는 output behavior가 더 신뢰성 있는 것으로 나타났습니다.

현재 프로젝트 판정의 유효성을 정리하면, vocab_size가 명확히 다른 모델들인 Solar, A.X-K1, VAETKI, K-EXAONE의 판정은 유효합니다. vocab_size가 유사한 HyperCLOVAX는 추가 검증을 권장합니다.

## Q17: LLM-DNA 같은 functional fingerprint 방법을 운영할 때 어떤 함정을 조심해야 하나요?

**질문 시각**: 2026-04-30

**답변**:

본 프로젝트의 LLM-DNA 검증을 9개 모델 규모로 직접 운영하면서 학술 논문에는 등장하지 않는 *운영적* 함정을 네 가지 만났습니다. 어느 하나도 알고리즘 자체의 결함은 아니지만, 결과의 진위가 silent하게 망가질 수 있는 위험이 있어 후속 검증자에게 경고하는 의미로 정리합니다.

첫 번째 함정은 **응답 기반 fingerprint의 silent failure**입니다. Q15에서 설명했듯이 LLM-DNA 0.2.x의 기본 추출기는 모델이 생성한 텍스트를 외부 sentence-encoder로 임베딩합니다. 만약 모델이 빈 응답을 내거나 generation 단계에서 RuntimeError로 죽으면, sentence-encoder는 빈 문자열에 대한 *fallback embedding*을 반환합니다. 그러면 모든 빈 응답 모델이 동일한 fallback 벡터로 collapse하고 거리 행렬에서는 0.0인 것처럼 보입니다 — 실제로는 아무 정보가 없는데 마치 형제 관계인 것처럼. 본 프로젝트에서 처음 1차 추출 결과 Solar와 Mixtral-base 거리가 0.0으로 나와 "이 둘이 같은 부모인가?"라는 의심이 들었지만, `responses.json`을 열어보니 두 모델 모두 100/100 빈 응답이었던 것이 정체였습니다. 방어책은 분석 전에 항상 `mean(len(item.response)) > 100` 같은 sanity check를 통과시키는 것입니다. 이 단순한 한 줄이 가짜 0.0 페어를 모두 잡아냅니다.

두 번째 함정은 **GPU 세대와 transformers 버전의 호환성 매트릭스**입니다. transformers 5.7.0은 Mixtral 같은 MoE 모델의 forward에서 `torch._grouped_mm` 연산을 자동 사용하는데, 이 연산은 CUDA compute capability 9.0 (Hopper: H100, H200) *전용*으로 설계됐습니다. L40S(8.9), A100(8.0), A10G(8.6) 등 그 이전 세대 GPU에서는 `RuntimeError: torch._grouped_mm is only supported on CUDA devices with compute capability = 9.0`이 발생합니다. 본 프로젝트가 Solar와 Mixtral-Instruct를 처음에 g7e 인스턴스(L40S)에서 돌렸을 때 100/100 generation이 RuntimeError로 죽었는데, `--continue-on-error` 옵션 때문에 잡 자체는 *Completed* 상태로 끝나고 빈 응답만 출력에 남아있었습니다. 방어책은 두 가지인데, 모델이 MoE이거나 transformers 5.x를 강제하면 인스턴스를 H100/H200으로 못 박거나, 아니면 transformers 4.x로 다운그레이드하는 fallback 경로를 entrypoint에 두는 것입니다.

세 번째 함정은 **multi-GPU 분산의 통신 오버헤드**입니다. 100B+ 모델이나 fp16 94GB Mixtral은 단일 GPU 메모리에 들어가지 않아 분산이 필수지만, `device_map="auto"`로 8 GPU에 layers를 *균등 분산*시키면 매 token forward마다 7번의 GPU 간 NCCL 호출이 발생해 generation 속도가 5-10배 느려집니다. 본 프로젝트의 Mixtral-8x7B-Instruct on p5.48xl 잡이 prompt당 30-60초 걸리는 비정상적으로 느린 속도를 보였는데, 이는 active params가 ~13B에 불과한 모델을 7개 GPU에 쪼개 통신 비용이 forward 비용을 압도한 결과입니다. 더 좋은 선택은 *모델이 들어가는 최소 수의 GPU*에만 분산하는 것이지만, llm-dna ModelWrapper의 기본 동작은 이를 지원하지 않습니다. 방어책으로는 인스턴스 선택 시 모델 크기에 맞춰 GPU 수를 줄이거나(예: Mixtral 47B는 2×H100 100GB로 충분), `device_map`을 수동 지정하는 것입니다.

네 번째 함정은 **`--continue-on-error`의 양면성**입니다. `calc-dna`의 이 옵션은 일부 prompt 처리 실패 시에도 잡을 끝까지 진행시켜 부분 결과라도 건지자는 의도지만, *모든* prompt가 실패해도 잡은 정상 종료합니다. SageMaker는 이를 `Status: Completed`로 표시하고 spot 재시도도 발생하지 않습니다. CloudWatch 로그를 직접 들여다보지 않으면 100건의 `Text generation failed` 에러가 보이지 않고, 외부에서 보면 잡이 성공한 것처럼 보입니다. 방어책은 entrypoint(`dna_train.py`)에서 calc-dna 종료 후 `responses.json`을 검증해 빈 응답 비율이 임계값(예: 10%)을 넘으면 non-zero exit code로 종료하는 것입니다. 그러면 SageMaker가 잡을 *Failed*로 표시하고 spot 재시도까지 받을 수 있습니다.

이 네 함정의 공통점은 모두 *silent*하다는 것입니다. 알고리즘이 깨지지 않고, 잡이 죽지 않고, 결과 파일이 정상 형식으로 출력됩니다. 그래서 분석 단계에서 거리 행렬과 트리만 보고는 함정의 존재를 알 수 없습니다. 본 프로젝트가 이 함정들을 모두 잡아낸 것은 결과가 직관과 충돌했기 때문입니다 — "Solar가 Mixtral-base와 거리 0.0이라고? 말이 안 된다." 그 의심이 sanity check로 이어지고, sanity check가 silent failure의 정체를 드러냈습니다. 기계적 검증 파이프라인을 만들 때는 *직관에 반하는 결과를 자동으로 의심하는 단계*를 명시적으로 두어야 한다는 교훈입니다.

학술 논문은 알고리즘의 정당성을 증명하는 데 집중하고 운영 함정은 다루지 않는 경향이 있습니다. 후속 검증자가 LLM-DNA 또는 유사한 functional fingerprint 방법을 적용할 때 본 프로젝트의 네 가지 함정과 방어책을 미리 인지하고 있다면, 본 프로젝트가 디버깅에 쓴 시간(약 2일)을 크게 단축할 수 있습니다.
